# v1.1 sketch — ideas worth landing next

Captured from comparing v1 against three adjacent projects:

- **[zebbern/claude-code-discord](https://github.com/zebbern/claude-code-discord)** — server-hosted Claude bot with rich slash-command UX
- **[JessyTsui/Claude-Code-Remote](https://github.com/JessyTsui/Claude-Code-Remote)** — multi-platform notification + tmux-injection bridge
- **[ebibibi/claude-code-discord-bridge](https://github.com/ebibibi/claude-code-discord-bridge)** — Python supervisor that spawns one `claude -p` per Discord thread

Each idea has a *why*, *cost*, and *priority*. Priority scale:

- **P0** — would ship in v1.1 unless there's a reason not to
- **P1** — likely v1.2; worth keeping in mind during v1.1 design so we don't paint into a corner
- **P2** — speculative; capture so we don't reinvent later
- **❌** — explicitly not doing; recorded with reasoning so we don't reconsider blindly

---

## Quick wins (P0)

### 1. Branch-aware thread names

**From zebbern.** Currently `<cwd-basename>-<short-id>`. Add the active git branch when there is one:

```
my-project-main-a1b2c3        # branch="main"
my-project-fix-auth-a1b2c3    # branch="fix-auth"
my-project-a1b2c3             # not in a git repo
```

**Why:** Working three branches of the same repo today produces three threads named `my-project-XXXXXX` and you can't tell which is which without opening them.

**Cost:** ~10 lines in `src/session-id.ts`. New `deriveThreadName(cwd, sessionId, { branch?: string })` overload, sanitize branch (Discord allows letters/digits/dashes/underscores, slashes commonly used in branches need substitution).

**Caveats:** If the user changes branch mid-session the thread name is stale. Don't rename — the thread persists; the name records "what branch this session started on." Document this.

### 2. `DISCORD_THREAD_NAME` override

**From zebbern.** Honor an env var that overrides auto-naming entirely.

```sh
DISCORD_THREAD_NAME="prod-incident review" \
  DISCORD_THREAD_ID=auto \
  claude --channels plugin:discord@danielfbm
```

**Why:** "hotfix lobby" beats "my-project-a1b2c3" every time for shared-team threads.

**Cost:** ~5 lines in `src/shim.ts`. Pass `name_hint` through `register` → daemon prefers it over `deriveThreadName(cwd, session_id)`.

**Caveats:** Truncate to 100 chars. Reject empty string after trim.

### 3. `/discord:bind <thread_id>` skill

**From zebbern's `/claude-thread`, deferred from v1.** Attach a running CC session to an existing Discord thread without restarting.

The skill writes to `bindings.json` directly (the daemon re-reads on next register). For a *currently running* session to pick up the binding, the user has to restart the shim — which means restart CC. Acceptable for v1.1; a richer "live rebind" RPC is P1.

**Cost:** New skill file `skills/bind/SKILL.md` — ~50 lines. Modifies `bindings.json` only; no daemon code.

**Note:** Verify the thread's parent is in `groups` before binding (mirror the daemon's `thread_not_allowed` check). Skill should fail loudly otherwise.

### 4. `/discord:list` skill

**From the `/discord:status` pattern.** Print active sessions, their threads, the DM session, and the daemon PID.

Reads `daemon.pid`, `bindings.json`, and asks the daemon (via UDS one-shot) for live session list. If daemon isn't running, just print bindings + "daemon not running."

**Cost:** Skill file ~30 lines. Maybe a daemon `list_sessions` UDS message — ~20 lines in `src/daemon.ts`.

### 5. Stop / SubagentStop hook auto-notify

**From JessyTsui.** Their best feature. When CC's `Stop` hook fires (agent finished a turn) and the agent didn't send a Discord message recently, push a "task complete" ping to the bound thread.

We ship the hook as **opt-in** via a config flag (`access.json: notifyOnStop: true`):

```
~/.claude/settings.json hooks:
  Stop: ["bash -c 'curl --unix-socket ~/.claude/channels/discord/daemon.sock http://x/notify-stop?session=$CLAUDE_SESSION_ID'"]
```

Or, more honestly: have `/discord:configure stop-hook on` write the hook into the user's settings.

**Why:** Long-running agentic tasks where the agent doesn't think to send a chat message at the end. User on phone gets pinged.

**Cost:** Modest. Daemon HTTP-over-UDS endpoint (~30 lines), hook command, skill arm, config flag. Need to map session_id → bound thread → post to it.

**Caveats:** If the agent already sent a reply in the last N seconds, suppress — don't double-notify. CC's `Stop` hook payload includes session info; need to verify what env vars are exposed.

### 6. Permission prompt "See more" persistence across daemon restart

**From v1's known limitation.** Today, `pendingPermissions` is in-memory. If the daemon restarts mid-prompt, "See more" responds with "details no longer available."

**Cost:** Persist `pendingPermissions` to `~/.claude/channels/discord/permissions.json` on every set/delete. Load at boot. Trivial.

**Why:** Already documented as a wart. Cheap to fix.

## Bigger ideas (P1)

### 7. Worktree-per-session isolation

**From ebibibi.** Optional safety property: each thread session runs in its own git worktree, derived from the bound branch. Two CC sessions on the same branch can't step on each other's working tree.

**Why:** Concurrent agentic edits on the same repo are a real footgun. Worktrees solve it cleanly.

**Cost:** Significant. Need a launcher script (or skill) that creates the worktree, then `cd`s and runs `claude` from there. Bindings need to track worktree path. Cleanup story (when do we remove worktrees?). User-level decision: opt-in at session creation, not retroactive.

**Open question:** Should this live in our plugin at all, or as a sibling skill? Worktree management is independent of Discord. Probably a separate skill.

### 8. MCP elicitation for permission prompts

**From ebibibi.** Use MCP's standardized elicitation API instead of our custom `claude/channel/permission` capability. Cleaner long-term, alignment with the broader MCP ecosystem.

**Why:** Future-proof. The Anthropic-internal `claude/channel/permission` is experimental; standard MCP elicitation is in the spec.

**Cost:** Investigation needed — is elicitation on by default in CC? Does it support our flow (request from MCP server → external response → result)? Probably a v1.2 thing; track upstream.

### 9. Persistent transcript / session resume metadata

**From RichardAtCT/claude-code-telegram (SQLite session persistence).** Our `bindings.json` holds thread mapping. We don't persist the *conversation* — Discord does that. But we don't track *when CC last said something* or *the rolling context size*. Adding lightweight session-level metadata (last_activity_at, message_count, working_dir_state) would let `/discord:list` show useful columns and enable smarter idle-exit / archival.

**Cost:** Extend `bindings.json` schema (forward-compat). Daemon writes on each inbound/outbound. Small.

**Why:** Better `/discord:list` UX, foundation for archival rules.

### 10. Live rebind RPC

**Extension of #3.** Today, `/discord:bind` only takes effect on next CC restart. A live RPC would let a running session change threads without a restart — useful when you accidentally start a session unbound and want to move it into a thread.

**Cost:** Daemon-shim protocol gains a `rebind {thread_id|null}` message. Shim translates to a daemon-side route swap. Carefully — there's a window where messages could leak; need to drain in-flight first.

**Priority:** Only after #3 is shipped and we see real demand.

### 11. Telegram channel sibling

**From JessyTsui.** Same architecture (daemon + shim) but Telegram instead of Discord. Most of our code is platform-agnostic (gate, access, framing, protocol) — only `discord-ops-real.ts` and `daemon-entry.ts` are Discord-specific.

**Cost:** Real but bounded. Refactor `DiscordOps` → `ChannelOps`; new `TelegramOps`; new `daemon-entry-telegram.ts`. Probably belongs in a separate plugin/repo, not this one.

**Why:** Telegram has better mobile UX than Discord for some users. RichardAtCT has 2.5k stars in the Telegram niche; demand is real.

**Note:** Discord has buttons; Telegram has inline keyboards (similar). Most of the permission UX translates cleanly.

## Speculative (P2)

### 12. First-message-based thread renaming

**From zebbern's UX.** After the user's first real message (not a "hi"), summarize it into a 5-word thread title. One LLM call inside the daemon.

**Why:** "my-project-a1b2c3" → "fix login redirect bug" tells the story.

**Cost:** Daemon needs an Anthropic API key (or Claude Code SDK) — we explicitly don't require this in v1. Adds a deployment dependency.

**Decision:** Defer. Auto-name is good enough; if users want better names, override (#2) covers it.

### 13. Channel monitoring auto-investigate

**From zebbern.** Watch a configured channel for messages from specific bots/webhooks (e.g. PagerDuty, Sentry alerts). When one arrives, auto-spawn a thread, ping Claude, paste the alert as the first message.

**Why:** "Sentry posted in #alerts → Claude in a thread starts investigating without me asking" is a powerful pattern for incident response.

**Cost:** Substantial. We don't *spawn* CC processes — that's the user's job. We could post a "Claude, investigate this?" prompt-template message and let the user trigger CC manually. But the auto-launch part requires CC supervisor capability we explicitly punted on.

**Decision:** Out of scope unless we add a CC supervisor mode (which would be a different project — closer to ebibibi or zebbern).

### 14. Multi-bot per host

Already supported via `DISCORD_STATE_DIR`. But the daemon-spawn logic in the shim assumes one daemon per host. If a user runs two bots (different tokens, different state dirs), each launches its own daemon — already works. Document it more clearly.

### 15. Multi-machine support

Punted explicitly in v1. The daemon-shim protocol is local UDS. To go cross-host:

- Replace UDS with TLS-authenticated WebSocket
- Add a daemon discovery mechanism (`DISCORD_DAEMON_URL`)
- Stream attachments instead of passing paths
- Persist auth secret out-of-band

**Decision:** Real demand will tell us if v2 is justified. For now, single-host is the right scope.

## Things explicitly *not* doing (❌)

These came up during the comparison; recording the reasoning so we don't relitigate.

### ❌ Run Claude inside the bot

zebbern + RichardAtCT both do this. It's a different product — a hosted Claude service. Our value prop is "your local CC, made remote." Hosting Claude requires:

- API key on the bot host
- Working dir / git context inside the bot
- File checkpointing, sandbox config, RBAC
- Different audience entirely

If we wanted that, we'd fork zebbern, not extend ours. They've already built it.

### ❌ tmux/PTY output scraping

JessyTsui's whole approach. Pre-MCP-era engineering. Output parsing is fragile (terminal escape codes, multi-line edits, redraws). The `notifications/claude/channel` MCP capability exists *because* this approach is bad. We use the right tool.

### ❌ 45+ slash commands

zebbern's surface. Discord becomes the IDE. We don't want this — Claude Code IS the command surface; our role is to relay messages, not replicate `/git`, `/shell`, `/rewind`, etc. inside Discord.

### ❌ Email or LINE channels

JessyTsui has these. They're a different audience (notify-on-completion vs. live chat). If we add a sibling channel, it's Telegram (#11), not email.

### ❌ RBAC via Discord roles

zebbern has this; we have allowlist-by-user-snowflake. Roles are a v3 thing if we ever ship in environments with team admins. v1.1 is for individual power users.

## Suggested v1.1 scope

Pick from P0 only. Ship narrow.

1. Branch-aware naming (#1) — 1 hour
2. `DISCORD_THREAD_NAME` env override (#2) — 1 hour
3. `/discord:bind <thread>` skill (#3) — 2 hours
4. `/discord:list` skill (#4) — 2 hours
5. Stop hook auto-notify (#5) — half day, bigger because it touches CC settings
6. Permission persistence across daemon restart (#6) — 1 hour

Rough total: a day's work. Each is independent — can land in any order, can stop after any.

P1 items (#7-#11) defer to v1.2 unless something specific motivates moving one up.
